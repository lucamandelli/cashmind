# 0009 — Theme toggle (light / dark / system)

## Status

accepted

## Context

ADR [[0005-design-token-system]] wired both light and dark themes into
`apps/web/src/index.css` (`:root` for light, `.dark` for dark) but explicitly
deferred the toggle UI: *"A theme toggle is now cheap but out of scope."*

The Welcome and Login screens are now considered stable (no further redesigns
until the full API is complete). A Claude-designed reference for both screens —
including a three-way toggle — exists in `design-system/`. This is the right
moment to ship the toggle: the foundation is already built, and doing it now
means every future screen inherits a working `useTheme` hook and `ThemeToggle`
component rather than having to set it up themselves.

## Decision

### Preferences: `light | dark | system`

Three-way, not two-way. `system` (the **default**) follows
`prefers-color-scheme` and re-applies when the OS setting changes at runtime.
`light` and `dark` override the OS. Stored as the string `"light"`,
`"dark"`, or `"system"` in `localStorage["cm-theme-pref"]`.

### Application: `.dark` class on `<html>`

The resolved theme is applied by toggling the `.dark` class on
`document.documentElement`. This is exactly the strategy `index.css` and all
`dark:` Tailwind utilities already assume (ADR 0005, §4). The design file's
`data-theme` attribute is not adopted — using it would diverge from `.dark` for
no gain.

### No-flash inline script in `index.html`

A tiny blocking `<script>` in `<head>` reads `localStorage["cm-theme-pref"]`
and `matchMedia` before React mounts. Without it, users who saved a dark
preference see a white flash on every load. The Welcome screen is the public
front door; a flash there is a poor first impression.

### Scope: auth screens only (accounts intentionally not guarded)

The theme foundation is **global** — `useTheme` toggles `.dark` on `<html>`,
so the choice flows into the entire app. The post-login app (accounts) is a
**throwaway WIP** that will be replaced by a Claude-designed rebrand once its
API is complete. That rebrand will inherit `useTheme` and `ThemeToggle` as-is.
No special guarding of the authenticated shell is needed or desirable.

### `ThemeToggle` component

A sun / monitor / moon segmented radiogroup (`role="radiogroup"`), fixed to
the top-right corner of the viewport. Styled exclusively with design-role
tokens (`bg-surface`, `border-border`, `text-primary`, `bg-primary-subtle`,
`text-text-3`, focus via `--focus`) — no hard-coded colour values.

### Design file is reference, not drop-in

`design-system/theme.tsx` (generated by Claude's design tool) uses a
`data-theme` attribute + an inline `<ThemeTokens>` `<style>` block with
`--cm-*` custom properties. Neither is adopted: our tokens already live in
`index.css` and our strategy uses `.dark`. The design's visual intent (toggle
shape, grain/glow behaviour, token wiring) is faithfully reproduced, but the
implementation is adapted to our architecture.

## Alternatives considered

- **`data-theme` attribute (from the design file).** Rejected: diverges from
  `.dark` and all existing `dark:` utilities; would require a separate Tailwind
  variant declaration.
- **Scope toggle to auth screens only (don't put `.dark` on `<html>`).** Rejected:
  weakens the foundation — future screens would have to re-wire separately.
  The authenticated WIP being temporarily off-palette in dark is an acceptable
  and short-lived trade-off.
- **Two-way toggle (light / dark only).** Rejected: `system` is the correct
  default for an app that respects OS preference. Omitting it would force users
  to pick manually on first visit.

## Consequences

- `useTheme` (`hooks/useTheme.ts`) and `ThemeToggle`
  (`components/ThemeToggle/`) are now the **canonical** theme mechanism for the
  app. Every future page that wants a toggle imports them — no new wiring.
- The no-flash script must stay in `<head>` in `index.html`. Do not move it
  below the stylesheet or defer it — the whole point is it runs before paint.
- `localStorage["cm-theme-pref"]` is the persistence key. If it is absent, the
  resolved theme is always `system` (no dark class added unless OS is dark).
- The accounts screen will appear with its current hardcoded light colours
  when dark mode is active until its rebrand lands. This is intentional and
  documented.
